# Пайплайн генератора, артефакты и проверки

Описание соответствует корневому `README.md` и коду в `codegen/`.

## Что делает веб-приложение генератора

- Пользователь задаёт **базовый каталог**, **имя проекта**, **профиль** и **задание (task)**.
- Перед генерацией выполняется **анализ сложности** задачи (общая оценка, риски по этапам, опции). Модель настраивают так, чтобы **не дублировать** советы про Docker/README/OOP — см. [llm-behavior.md](llm-behavior.md).
- Создаётся каталог **`<base>/<project-name>`** (без суффикса с датой).
- Запускается **пайплайн** (в логах помечается как **[1/4] … [4/4]**):
  1. **Архитектура** — JSON-план слоёв, классов, файлов.
  2. **Скаффолд** — JSON со списком файлов и содержимым.
  3. **Ревью** — JSON с `issues`, `corrected_files`, `approved`; правки подмешиваются в файлы.
  4. **Постобработка** — правки `requirements.txt`/README (язык, структура, HTTP/TLS), автодозаполнение обязательных корневых файлов при отсутствии, **stack checks**, финальный `_guard`.

Подробные системные промпты и постобработка README — в [llm-behavior.md](llm-behavior.md).

- Пишутся артефакты и **логи в UI в реальном времени (SSE)**.

## Артефакты в генерируемом проекте

В типичном выводе (в каталоге проекта):

- `artifacts/architecture.json`
- `artifacts/scaffold.json`
- `artifacts/review.json`

## Stack checks (кратко)

Реализация: `codegen/stack_checks.py`.

1. **Согласованность `openai` SDK**  
   Если в `requirements.txt` указан пакет `openai`, стиль кода (legacy `ChatCompletion` / `.acreate` vs клиент `OpenAI` / `AsyncOpenAI` / `chat.completions`) должен соответствовать заявленной мажорной версии. При несоответствии — **ошибка** с понятным текстом.

2. **Плотность `pass` в продакшен-коде**  
   Подсчёт голых `pass` в **не-тестовых** `.py`. Если число ≥ порога из `defaults.yaml` (`stub_pass_warning_threshold`, по умолчанию 14), в лог пишется **предупреждение** о возможных незавершённых заглушках.

Исключения для тестов: пути с `/tests/`, `tests/`, имена `test_*.py`, `*_test.py` не учитываются в эвристике `pass`.

## Маппинг каталогов (Docker UI → хост)

В типичной конфигурации `docker-compose.yml` этого репозитория:

- На хосте: например `/home/dns/projects`
- В контейнере backend: `/workspace/projects`

В UI значение по умолчанию для базового каталога часто **`/workspace/projects`**, что соответствует выбранному пути на хосте после монтирования тома (см. ваш актуальный `docker-compose.yml`).
