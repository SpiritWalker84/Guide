# Анализ ТЗ перед реализацией

Правило для ассистента: **перед написанием кода** не бросаться сразу в реализацию, а сначала разобрать задание так, как если бы вы вели архитектурную сессию с заказчиком.

## 1. Детальный разбор ТЗ

- Выпишите **цель**, **границы scope** (что точно входит и что явно вне задачи).
- Уточните, **нужен ли запуск через Docker Compose** (`docker compose up -d`): для сервисов и многосоставных приложений — по умолчанию **да**; для простого CLI или односкриптового задания — можно **нет** (см. [docker-and-compose.md](docker-and-compose.md)).
- Зафиксируйте **входы/выходы**, **интеграции**, **нефункциональные ожидания** (производительность, безопасность, доступность, форматы данных), если они следуют из текста или разумно вытекают.
- Отметьте **неясности, противоречия и пробелы** в формулировках — отдельным списком.

## 2. Сложности и риски

- Кратко опишите **технические и продуктовые риски** (зависимости, масштаб, краевые случаи, миграции, совместимость).
- Если чего-то не хватает для уверенной реализации — **явно скажите**, чего не хватает, и что можно сделать по умолчанию при молчании заказчика (и какие допущения вы при этом принимаете).

## 3. Варианты реализации — в виде вопросов с A) B)

Когда допустимо **несколько разумных способов** реализации (стек, архитектура, хранение данных, API, очередь vs синхронный вызов и т.п.):

- Сформулируйте **вопрос к пользователю** (один вопрос — одна тема решения).
- Для каждого варианта используйте обозначения **A)**, **B)**, при необходимости **C)** и далее.
- Под каждым вариантом кратко укажите **плюсы** и **минусы** (по 2–5 пунктов, по делу, без воды).
- Если есть **рекомендация по умолчанию** — назовите её и объясните в одном предложении, почему.

Пример структуры (шаблон):

```text
Нужно уточнение по хранению состояния:

A) Redis — плюсы: …; минусы: …
B) PostgreSQL — плюсы: …; минусы: …

Рекомендация по умолчанию: A), если уже есть Redis в инфраструктуре; иначе B).
```

## 4. Когда переходить к коду

- После анализа: если были вопросы с вариантами — **дождитесь выбора** пользователя (или явного «делай по варианту B» / «сам выбери с обоснованием»).
- Если вопросов нет или ТЗ однозначно — можно сразу перейти к плану работ и реализации, **соблюдая остальные документы** в `docs/conventions/`.
- После заметной реализации выполните **код-ревью по чеклисту** — см. [code-review.md](code-review.md).

## Язык

Формулировки вопросов, плюсов/минусов и резюме анализа — **на русском**, в том же деловом стиле, что и docstrings в [code-style-and-comments.md](code-style-and-comments.md).
