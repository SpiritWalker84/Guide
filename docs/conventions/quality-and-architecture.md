# Качество и архитектура

Правила из `defaults.yaml` (`quality`, `use_oop`, `modular_structure`) и профилей.

## ООП и структура

- **`use_oop: true`** — проекты ориентированы на объектно-ориентированный стиль: классы для сервисов, конфигурации, инициализации приложения (конкретика зависит от профиля).
- **`modular_structure: true`** — чёткие границы модулей, не «мешанина» в одном файле.
- **`include_tests: true`** — по умолчанию предусматриваются тесты (каталог `tests` в стандартной структуре).

## Качество (`quality`)

- **clean_code** — чистый, поддерживаемый код без лишней сложности.
- **performance_first** — при прочих равных приоритет производительности и осмысленной работы с ресурсами (без превращения в преждевременную оптимизацию «ради красного словца»).
- **require_modular_interactions** — взаимодействие модулей явное, через устойчивые интерфейсы и контракты, а не через скрытые глобальные связи.
- **prefer_composition: true`** — по возможности композиция вместо глубокого наследования. **Исключение:** профиль `cli-tool` выставляет `prefer_composition: false` (допускается иной расклад для CLI).

## Стиль кодирования (`coding_style`, обобщённо)

- Ясные границы модулей и **небольшие классы** с понятной ответственностью.
- **Композиция предпочтительнее наследования**, где это уместно (см. исключение для `cli-tool` выше).
- Секреты и чувствительные значения **не хардкодить** — только переменные окружения (см. [environment-and-secrets.md](environment-and-secrets.md)).
- Код **читаемый, поддерживаемый**, с упором на предсказуемое поведение и при необходимости — производительность.
- **Явное проектирование** взаимодействий между модулями (стабильные интерфейсы).

## Базовый профиль `base`

- Слоистая, модульная архитектура.
- Мелкие классы с единственной ответственностью (SRP).
- Секреты и runtime-значения из окружения.
- Dockerfile ставит зависимости из `requirements.txt` и запускает приложение.
- `docker-compose.yml` — один сервис приложения на свободном порту.

## Запуск в продакшен-подобном окружении

Для типичных сервисных проектов ожидается **Docker Compose** и команда **`docker compose up -d`** как основной способ поднять всё согласованно — см. [docker-and-compose.md](docker-and-compose.md). Простые CLI и мелкие скрипты могут обходиться без compose.

## Стандартные каталоги (`standard_dirs` в defaults)

По умолчанию в скелете учитываются:

- `src`
- `tests`
- `config`
- `scripts`
- `artifacts`

У конкретных профилей список может расширяться (например `src/api`, `src/services` и т.д.) — см. [profiles.md](profiles.md).
