# Стиль кода, комментарии и docstrings

## Язык комментариев и docstrings

- **`generation_policy.code_comments_locale_default: ru`** — если в техническом задании **не указан другой язык** для документации внутри кода, **комментарии и docstrings в исходниках — на русском**, деловой профессиональный стиль.
- То же закреплено в `coding_style` и `prompt_rules`: модули, классы, публичные методы, нетривиальная логика должны быть документированы **без воды, без транслита**; объяснять **зачем**, контракты и неочевидные решения.

## Что документировать

- Публичные API (модули, классы, методы/функции с нетривиальным поведением).
- Неочевидные инварианты, ограничения, побочные эффекты.
- Интеграции (внешние API, форматы данных) — кратко и по делу.

## Общий тон кода

- **Простой, ясный код** важнее «умных» конструкций: понятные имена, короткие функции, предсказуемый поток.
- **Малые классы**, **чёткие границы модулей** (см. [quality-and-architecture.md](quality-and-architecture.md)).
- Если в задании **явно** просят другой язык для комментариев/docstrings — следовать заданию.

## README пользователя проекта

Язык **README сгенерированного** проекта — **русский** (отдельное требование к артефакту README, не путать с комментариями в коде). Структура разделов — в [readme-onboarding.md](readme-onboarding.md).

## Дословно: `coding_style` из `defaults.yaml`

- Use clear module boundaries and small classes.
- Prefer composition over inheritance when possible.
- Never hardcode secrets; use environment variables.
- Prioritize readable, maintainable, and high-performance code.
- Design module interactions explicitly via stable interfaces.
- Если в задании явно не просят другой язык: docstrings и комментарии в коде — на русском, профессионально (модули, классы, публичные методы, нетривиальная логика); без воды, без транслита, объясняйте «зачем» и контракты там, где это неочевидно.
