# Код-ревью

Здесь два смысла: **ревью внутри генератора** (автоматический шаг пайплайна) и **ревью, которое должен делать ассистент**, когда вы просите код «по этому гайду» без самого генератора.

---

## 1. Ревью в проекте ProxyAI Code Generator

После генерации файлов пайплайн вызывает **строгий LLM-ревью** (`_review_chain` в `codegen/pipeline.py`):

- На вход подаётся полный набор сгенерированных файлов и правила (`quality`, `generation_policy`, архитектурный план).
- На выходе — JSON: `approved`, `issues` (severity **high** / **medium** / **low**), `corrected_files` (полные тексты исправленных файлов).
- **Высокий** приоритет: исправления, которые можно сделать правкой файлов, должны попасть в `corrected_files`, а не только текстом в `issues`.
- Если после правок проект по-прежнему нельзя считать запускаемым — ревьюер выставляет **`approved: false`** (стоит проверить `artifacts/review.json`).

Что именно проверяет ревьюер (смысл промпта):

- Модульность, OOP, контракты между модулями, ориентация на производительность.
- Согласованность **версий в `requirements.txt`** со стилем использования библиотек в коде (мажорные линии API).
- Отсутствие «пустой» продакшен-логики: только `pass`, голый TODO, `NotImplementedError` на критичных путях.
- Регистрация маршрутов/обработчиков/сервисов из **точки входа / startup**, чтобы приложение могло стартовать.
- **Docstrings и комментарии** — в соответствии с политикой языка (по умолчанию русский), покрытие публичных API и неочевидной логики.
- **README.md** — разделы из `readme_onboarding` (цель, стек, структура, запуск, env, разработка/тесты).

После подмешивания правок идут **постобработка**, **stack checks** и **`_guard`** — см. [pipeline-and-checks.md](pipeline-and-checks.md) и [llm-behavior.md](llm-behavior.md).

**Артефакт:** в сгенерированном проекте сохраняется `artifacts/review.json` — удобно для аудита, что именно нашёл ревьюер.

## 2. Поведение ассистента по гайду (вне генератора)

Если вы работаете в чате по [tz-analysis-workflow.md](tz-analysis-workflow.md) и остальным файлам `docs/conventions/`, после **существенной реализации** (новый модуль, нетривиальный рефакторинг, готовый PR-уровень изменений) ассистент должен:

1. **Провести саморевью** по тому же духу, что и `_review_chain`: пройтись по чеклисту ниже и либо внести правки в код, либо явно перечислить оставшиеся риски.
2. **Структурировать замечания** по серьёзности (высокая / средняя / низкая), как в `issues` генератора.
3. **Не считать задачу закрытой**, если есть проблемы **высокой** серьёзности (не запускается, секрет в коде, несовместимые версии зависимостей и API, сломанная регистрация маршрутов и т.п.) — их нужно устранить или согласовать с вами явно.

Опционально (если уместно по объёму): короткий блок «что бы сказал ревьюер» перед финальным ответом.

## 3. Чеклист ревью (для ассистента и для людей)

Сверка с правилами из [quality-and-architecture.md](quality-and-architecture.md), [generation-policy.md](generation-policy.md), [environment-and-secrets.md](environment-and-secrets.md), [readme-onboarding.md](readme-onboarding.md):

| Тема | Проверить |
|------|-----------|
| Запуск | Точка входа, bootstrap, зарегистрированы handlers/routes/клиенты; нет обрыва цепочки инициализации. |
| Заглушки | Нет оправданных только `pass` / пустых тел / `NotImplementedError` на продакшен-путях без явной договорённости. |
| Зависимости | `requirements.txt` (или аналог) с пинами; импорты и вызовы API соответствуют **мажорной** версии библиотек; нет смеси старых и новых паттернов одного SDK (см. OpenAI в [pipeline-and-checks.md](pipeline-and-checks.md)). |
| Секреты | Нет ключей и паролей в коде; только env / `.env.example` с описанием. |
| Архитектура | Слои и модули разделены; контракты между слоями понятны; композиция там, где это задано правилами профиля. |
| Стиль и документация | [code-style-and-comments.md](code-style-and-comments.md): русские docstrings/комментарии по умолчанию, без транслита. |
| README | Если менялся публичный контракт проекта — README обновлён по [readme-onboarding.md](readme-onboarding.md); упоминание HTTP по умолчанию при отсутствии TLS. |
| Docker | Если в scope есть compose — сервисы, порты, `env_file` согласованы с [docker-and-compose.md](docker-and-compose.md). |

После ревью при необходимости — прогон тестов/линтера, если они есть в репозитории (как в разделе «Разработка и тесты» README).
