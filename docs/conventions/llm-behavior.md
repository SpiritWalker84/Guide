# Поведение LLM: этапы, промпты и допущения

Сводка по `codegen/pipeline.py` и `codegen/task_analysis.py`. Ответы модели в JSON-цепочках ожидаются как **строгий JSON** (`response_format: json_object`, низкая температура в клиенте).

## Анализ задачи до генерации (`task_analysis.analyze_task`)

**System prompt (смысл):**

- Роль: senior solution architect.
- Вернуть только строгий JSON.
- Все текстовые поля ответа на **русском**: summary, этапы, риски, названия опций, плюсы/минусы.
- **Не предлагать** как этапы, риски или варианты: Docker, docker-compose, Dockerfile, контейнеризацию, Kubernetes — **генератор уже добавляет это** (кроме профилей без compose).
- **Не советовать отдельно** добавлять `.env.example`, `README.md`, `requirements.txt` или «общую модульную OOP-структуру» — это **уже принудительно**.
- Фокус анализа: доменная логика, интеграции, источники данных, каналы, расписание, граничные случаи, безопасность, НФТ под продукт.

**Допущения, передаваемые модели (`generator_pipeline_assumptions`, русский текст в коде):**

> Генератор проекта уже добавляет по умолчанию: docker-compose.yml, Dockerfile (по профилю), .env.example, README.md, requirements.txt, модульную OOP-структуру, артефакты анализа. Не предлагай эти вещи как отдельные этапы, риски или варианты реализации.

После ответа код может **фильтровать** шум: этапы/риски/опции, целиком сведённые к Docker/K8s без продуктового смысла, убираются или ослабляются (`_filter_noise_from_stages`).

## Цепочка 1: архитектура (`_architecture_chain`)

**System prompt (смысл):**

- Senior software architect, только строгий JSON.
- Спроектировать **модульную OOP-first** структуру с ясными контрактами модулей.
- **Композиция вместо наследования**, когда это требуют quality rules.
- Согласованный стек зависимостей: **мажорные версии в плане должны соответствовать реальным API библиотек** (не смешивать паттерны разных мажорных релизов).
- Дизайн должен быть **реализуем без «только заглушек»** в ядре.
- Все человекочитаемые строки в JSON (summary, ответственность классов, назначение файлов и т.д.) — **на профессиональном русском**, если в задаче явно не запрошен другой язык.

В user-payload передаются `task`, `project_slug`, `rules`, `quality_constraints`, `generation_policy` и схема полей (`layers`, `core_classes`, `file_plan`).

## Цепочка 2: генерация файлов (`_scaffold_chain`)

**System prompt (смысл):**

- Генерация **полных** файлов нового проекта, строгий JSON.
- Содержимое файлов — обычный текст, **готовый к продакшену**; код чистый, эффективный, с **явными** взаимодействиями модулей.
- **Реальное поведение:** не оставлять в handlers/services/repositories/startup только `pass`, голые TODO или `NotImplementedError` — минимальная рабочая логика или чётко ограничённые no-op, удовлетворяющие контрактам.
- **requirements.txt:** закрепить версии, совместимые с **каждым** импортом и API в коде (та же мажорная линия, что и в написанном коде).
- **Сборка приложения:** точка входа, composition root, регистрация маршрутов/обработчиков/клиентов по стеку — приложение должно **стартовать**.
- **Docstrings и комментарии:** если в задаче явно не требуется другой язык — **профессиональный русский** (модули, классы, публичные методы, нетривиальная логика); кратко, без воды и транслита; пояснять намерение и контракты.
- **README.md** — полноценный онбординг **на русском**, по разделам `readme_onboarding.required_sections_ru`, чтобы разработчик понял проект **без чтения каждого файла**.

В constraints для модели дополнительно: `oop_first`, `modular`, `env_in_dotenv_example`, обязательные корневые файлы (минимум `.env.example`, `README.md`, `requirements.txt`), пути относительно корня проекта.

## Цепочка 3: ревью (`_review_chain`)

**System prompt (смысл):**

- Строгий ревьюер, строгий JSON.
- Требовать: модульность, OOP, гигиену env, ориентацию на производительность, ясные межмодульные контракты.
- **Обязательно проверить:**
  1. Пины в `requirements.txt` согласованы со стилем импорта/API по **каждой** крупной библиотеке.
  2. Нет продакшен-путей, состоящих только из `pass`, голого TODO или `NotImplementedError`.
  3. Handlers/routes/services подключены из **entrypoint/startup**, приложение может запуститься.
- По каждой проблеме **высокой** серьёзности, исправимой правкой файлов — **обязательно** положить полное исправленное содержимое в `corrected_files`, а не только описание.
- Если после правок проект всё ещё не runnable — выставить `approved: false`.
- Docstrings/комментарии должны соответствовать `generation_policy.code_comments_locale_default` (по умолчанию русский): тон, покрытие публичных API и неочевидной логики.
- **README.md** — разделы из `readme_onboarding.required_sections_ru`: цель, стек, структура, запуск, переменные окружения, разработка/тесты при необходимости.

## Постобработка после ревью (`_postprocess_generated_files`)

- **Совместимость Flask/Werkzeug:** если в `requirements.txt` указан старый Flask (&lt; 2.3), строка может быть поднята до современной (в коде — до `Flask==3.1.0` и учёт совместимости).
- **README на русском:** если кириллицы нет — подставляется **шаблон README на русском** с блоками О проекте / Стек / Структура / Запуск / .env / Разработка и тесты / чеклист по `required_sections_ru` / примечание HTTP vs TLS.
- **Тонкий README:** если разделов второго уровня мало и нет явного «структур…» — добавляется секция **«Структура репозитория»** с автосводкой по сгенерированным путям и напоминанием дополнить «О проекте» и «Стек».
- **HTTP/TLS:** если в README нет явного упоминания HTTP и TLS/HTTPS — добавляется блок **«Примечание по доступу»** (по умолчанию HTTP, HTTPS только после настройки TLS, например reverse proxy).

## Финальная валидация (`_guard` и лимиты)

- Не больше **`MAX_FILES`** файлов в ответе.
- Каждый файл не больше **`MAX_FILE_BYTES`** байт (UTF-8).
- Обязательные файлы: `.env.example`, `README.md`, `requirements.txt`; если `include_compose` — ещё `docker-compose.yml`.
- Пути безопасные: не абсолютные, без `..` в компонентах пути.

После этого вызываются **stack checks** (OpenAI SDK, предупреждение по `pass`) — см. [pipeline-and-checks.md](pipeline-and-checks.md).
